## Overview

Material Design 3 is Google's current design system. Its visual language is **expressive, adaptive, and accessible**: rounded forms, tonal color, and a three-tier token architecture that allows a UI to respond to user color preferences at runtime.

The system is defined by four core decisions: a tonal palette-based color system, a five-role typescale, a named shape scale, and a semantic token layer that separates design intent from concrete values.

M3 is well-suited for: consumer apps, Google ecosystem products, cross-platform (Android/iOS/Web/Flutter), accessibility-first design, and any product that wants to support Dynamic Color or user-personalized theming.

## Colors

The color system is built on **tonal palettes** — for each key color, a full range from tone 0 (black) to tone 100 (white) is generated. From these palettes, semantic **roles** are assigned. Roles are what components use; raw palette values are never referenced directly in components.

The five key colors are Primary, Secondary, Tertiary, Error, and Neutral. Each generates a family of roles:

- **Primary** (`primary`, `on-primary`, `primary-container`, `on-primary-container`): The main brand color. High-emphasis interactive elements — filled buttons, FABs, active nav indicators.
- **Secondary** (`secondary`, `secondary-container`, etc.): A supporting accent. Less prominent than primary — filter chips, toggles, secondary actions.
- **Tertiary** (`tertiary`, `tertiary-container`, etc.): A contrasting accent for differentiation. Use sparingly for expressive or decorative moments.
- **Error**: Reserved exclusively for destructive or error states. Never repurpose these roles.
- **Surface family** (`surface`, `surface-container-*`, `on-surface`): Drives all backgrounds and planes. Uses tonal elevation — higher-numbered containers appear slightly more tinted, creating depth without shadows.

**The `on-*` pattern** is M3's primary accessibility mechanism. Every surface role has a paired `on-*` role for text and icons placed on it. These pairs always meet WCAG AA contrast (4.5:1 minimum). Never place content using a role that isn't the correct `on-*` partner for its background.

Dark theme uses the same role names with different tone assignments — primary-80 instead of primary-40, for example. Token names never change between themes.

## Typography

Two typefaces: **brand** (used for display, headline, title — expressive text) and **plain** (body, label — readable text). Both default to Roboto; replace via `--md-ref-typeface-brand` and `--md-ref-typeface-plain`.

Five roles: Display, Headline, Title, Body, Label — each in large/medium/small sizes. Role selection should follow content purpose, not desired visual size.

- **Display**: Hero moments only. One per screen maximum.
- **Headline**: Page and section titles.
- **Title**: Component-level headings — card titles, list item headers, app bar text.
- **Body**: Long-form readable content. Body Large for prose; Body Medium for standard descriptions.
- **Label**: All utility text — button labels, field labels, captions, chip text.

## Shape

Shape uses a named scale of seven steps: none (0px) → extra-small (4px) → small (8px) → medium (12px) → large (16px) → extra-large (28px) → full (pill).

Shape communicates component type and personality. Buttons and FABs are pill-shaped — inviting interaction. Cards are medium — balanced presence. Text fields are extra-small — functional and compact. Dialogs are extra-large — significant but approachable.

Reducing the scale globally (toward small/none) creates a more formal, utilitarian character. Keeping or increasing the defaults creates a friendly, expressive character.

## Components

- **Filled button**: Primary action. Use `primary` background with `on-primary` text. Pill shape.
- **Outlined button**: Secondary action. Transparent background, `primary` text and border. Pill shape.
- **Card**: Container surface. Use `surface-container` family. Medium corner radius.
- **Dialog**: Overlay container. `surface-container-high` background. Extra-large corners.
- **Chip**: Compact selection or filter. `surface-variant` background. Small corners.
- **Text field**: `surface-container-highest` fill. Extra-small corners. Label uses `on-surface-variant`; active indicator uses `primary`.

## Do's and Don'ts

**Do:**
- Generate your full color scheme from a single brand seed using the Material Theme Builder or `material-color-utilities`
- Always use `on-*` roles for text and icons — never place content using a surface role on its own background
- Use `*-container` roles for larger, lower-emphasis surfaces; use non-container roles for high-emphasis interactive elements
- Load all three Roboto weights (400, 500, 700) — or all three weights of any custom font
- Apply shape from the named scale — don't invent arbitrary pixel values

**Don't:**
- Use reference palette tokens (`--md-ref-palette-*`) directly in component styles
- Hard-code hex values in components — use system tokens
- Mix M2 class names or token conventions with M3 tokens
- Use Display or Headline typescale for body text
- Override all shapes to 0px unless a flat aesthetic is intentional and consistent across the system
